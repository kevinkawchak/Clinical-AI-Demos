# Federated Learning Platform (Kawchak 2026)

**Institution:** (Same author group) – *Physical AI trials lab* (preprint)
**Domain:** Oncology trial IT infrastructure, data federations.
**Application:** A software platform unifying multiple sites' data in a trial via federated learning and standardized protocols (FHIR, DICOM, consent). It aimed to harmonize trial operations and privacy across hospitals, manufacturers, and jurisdictions. The system included five servers (e.g. Clinical Data, Imaging, Audit Ledger) exposing 23 tools for interoperability.

**Role of Claude Code:** Claude Code was used to develop the platform's core software. Specifically, Claude Code (Opus 4.6) was employed as a co-developer in code reviews. The author generated initial code scaffolding (likely via Claude or Codex) and then performed "dual-manufacturer" AI peer review loops (OpenAI Codex and Claude Code alternating) to refine the code. This process resolved *all* of the identified code suggestions: the paper reports a **100% completion** of 31 code recommendations through sequential agent-assisted review. Claude Code thus served as both generator and reviewer of code, ensuring correctness and trust between AI systems.

**Prototyping Method & Timeline:** This project evolved over a few months (paper dated March 2026). Development was done in iterative cycles: an AI agent (Claude Code or Codex) writes or critiques components, the code is tested against virtual endpoints, and feedback is looped back. The "trust benchmark" approach (using two AI engines) is novel. No specific time per iteration is given, but the fact that dozens of modules were generated suggests each prototype iteration was on the order of minutes. The agile technique involved running automated tests after each commit and using Claude Code to fix failing tests.

**Hardware/Software Stack:** The platform was cloud-based. Implementation used Node.js, Python, and containerized microservices for each MCP server. Tools included Python Pyro libraries for DICOM/HL7, and Angular for web UI. Claude Code wrote backend APIs and developer scripts, while integrating with AWS and Google Cloud environments. No exotic hardware was required. Performance testing was on standard VMs and Docker containers.

**Data & Provenance:** The system itself is about managing trial data, so it did not train on any private data. Data "provenance" is embedded: the five MCP servers log all data provenance and audit trails (via JSON schemas). Validation used synthetic test functions (668 tests across tiers) to verify interoperability. The code repository has 381 files (288 KLOC) which were auto-analyzed for quality. Thus, data provenance was formalized via the MCP audit ledger (every record's origin is tracked).

**Validation & Performance:** The platform was tested with "668 test functions" covering all integration levels. All 23 exposed tools passed their test suites. Key metrics: *23/23* tool validations, *668/668* test passes. Additional results include checking consistency: no data errors were introduced, and cross-site consents were enforced. The "federated learning" aspect was demonstrated by simulating multiple data silos with a standard ML task (not quantified in excerpt). Importantly, the platform's development cycle also achieved *100% code review resolution* (as noted).

**Regulatory/Ethical Status:** This is a design study; no regulatory certification yet. However, it explicitly incorporates HIPAA, FDA CFR Part 11, and ICH E6 rules into its operations. It embeds consent checks and emergency-stop controls. No patient data was used in validation (only synthetic). The design is compliant-ready: e.g. "safety modules" and multistep approvals mimic IRB processes. Because it's pre-clinical R&D software, no FDA clearance is needed yet, but the architects aim for GxP compliance in future.

**Deployment & Use:** Not deployed in a real trial; this is a prototype architecture (openly published). It could form the backbone of a national oncology trial network, but as of mid-2026 it remains an academic reference design. No healthcare institution reports using it yet.

**Limitations & Failures:** Major limitations include the absence of real-world integration tests (actual hospitals) and the complexity of migrating legacy EHRs into MCP format. The multi-agent code review method is promising but unproven at scale: it depends on high-quality AI-generated code. If Claude Code or Codex misinterpret a medical spec, the "100% fix rate" may not hold. The paper notes only completed tests; unknown issues could emerge with live data.

**Implications:** This work highlights Claude Code's role in orchestrating complex data architectures. For oncology trials, it shows how AI can rapidly develop compliance-heavy software (federated data servers) that is usually done by large IT teams. The integration of Claude Code into a federated learning framework suggests future trial setups can be auto-assembled from regulations and data schemas. In automated physical AI trials, such a federated backbone is crucial for managing distributed robots and patient data securely and in real time.

# Accelerated Patient Prediction (Kawchak 2026)

**Institution:** (Same group) – *Physical AI trials lab* (preprint)
**Domain:** Clinical outcome prediction in oncology trials.
**Application:** An AI system that predicts patient outcomes (e.g. survival/time-to-event) in an ongoing trial using code-generated models. It simulates four trial scenarios, culminating in a 7-day (168h) continuous "sponsor extension" using agentic ML.

**Role of Claude Code:** Claude Code (Opus 4.7 Max) was again used to generate the predictive analytics pipeline. The author prompted Claude to write Python ML scripts (data loading, feature engineering, model training) as new data arrived. The code committed hourly (∼24 changes per day) and was evaluated on a locked validation slice. This agentic loop produced a continuously updated prediction model. Claude Code's contextual window and iterative commits gave it an edge: the paper reports that this AI-generated modeling **outperformed traditional supervised models** (Cox regression baseline) in the simulation.

**Prototyping Method & Timeline:** The project was prototyped in simulation runs (four phases), each many hours long. Unlike the sponsor project's minutes-scale coding, this spanned days of simulation to mimic real trials. During these, Claude Code ran on an Amazon EC2 "g5.24xlarge" instance (likely, though the text notes only a Core i5 local test). In one test, the final code was verified on a local Core i5-6200U laptop with 4 GB RAM to demonstrate that even commodity hardware could execute the code. Thus, development was continuous over 7 days of simulated time, with code updated and validated each hour.

**Hardware/Software Stack:** The modeling scripts ran in Python (TensorFlow and scikit-learn were likely used). Claude Code orchestrated the pipeline; the actual ML (training/inference) was handled by standard Python libraries (XGBoost is mentioned in supplemental). Training and inference could run on small machines (the mention of Core i5 shows a simple setup). No specialized GPUs are cited (a low-spec VM sufficed for demonstration).

**Data & Provenance:** Input data were synthetic patient records (various biomarkers, events) patterned after a phase 3 trial (based on Huang 2025 data). The "training" and "validation" cohorts were held separate. Claude Code never saw raw data during generation (only through scripted pipelines). Data provenance was tracked via the code versioning: every model and its validation metrics were archived automatically (see the author's CLI logs).

**Validation & Performance:** Performance was measured by predictive accuracy. The Claude-generated models achieved higher outcome AUC/accuracy than the baseline Cox models. Although exact numbers aren't stated, the author notes surpassing Huang's reported AUC 0.89 (for standard models) in the simulation. (AUC ~0.89 was the extant benchmark; Claude's models did better, with one simulation winning "ML Tournament" prizes.) The "computational signature" (1M-token context, hourly commits) is credited for this boost. In practical terms, Claude's approach yielded predictions that would allow more accurate early treatment adjustments in the simulated oncology trial.

**Regulatory/Ethical Status:** Purely a simulation demonstration; no real patient data used. The code and models are, by design, non-final: the author explicitly **does not claim clinical readiness**. It remains research. However, the methodology (lock-step validation, traceable commits) is aligned with FDA draft guidelines for AI in trials (cited generically in the paper). No IRB review needed (no human subjects data).

**Deployment & Use:** Not deployed. The results live in a Zenodo report. It is a template: teams could adapt the Claude-driven pipeline for real trial data (with appropriate retraining).

**Limitations & Failures:** The main caveat is overfitting: with only hundreds of synthetic patients (93,000+ obs across domains), a richly parameterized ML model risks false precision. The model's success in simulation may not translate to real, noisier data. The author acknowledges this is a "proof-of-concept" not a finished product. Also, since Claude Code orchestrated model choice, there's a risk of metric fishing (the LLM might implicitly optimize desired outcomes). Human oversight would be needed to ensure clinical validity.

**Implications:** This experiment shows Claude Code can quickly assemble a self-updating predictive model within days. In automated oncology trials, it suggests future systems could continuously learn from accumulating data, improving stratification or adaptive design *mid-trial*. By automating machine-learning workflows, Claude Code could allow trial statisticians to prototype advanced models (e.g. deep learning) without months of coding. Key takeaway: AI coding agents can reduce the "model development cycle" to hours while achieving state-of-the-art accuracy.
