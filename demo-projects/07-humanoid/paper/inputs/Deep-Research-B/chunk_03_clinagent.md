# ClinAgent (Yan 2026)

**Institution:** Harrisburg Univ. of Science & Technology (medRxiv preprint)
**Domain:** Clinical trial statistical programming (cardiovascular trial demo) – *applicable to oncology trials*
**Application:** *ClinAgent* is a software layer that equips any AI coding agent (Claude Code, etc.) with clinical-trial programming skills. It implements nine skills (from study setup to eSubmission packaging) for SAS/CDISC workflows. The goal is to automate table/listing/figure (TLF) generation for trial analysis.

**Role of Claude Code:** ClinAgent is model-agnostic ("Augment any agent"). In practice the author demonstrates it with Claude Code: Claude generates code fragments (SAS, SQL, R) guided by the nine skills. Crucially, *ClinAgent's MCP tools* handle data I/O (reading SAS7BDAT, Excel specs), so Claude only writes analytic logic. Thus Claude Code is the "assistant programmer" for trial stats.

**Prototyping Method & Timeline:** ClinAgent was developed as a software project with traditional coding and then integrated with Claude. The prototype and validation were completed by the Jan 2026 preprint date. The workflow was iterative: knowledge modules (skill templates) were hand-coded by Yan, while Claude Code was used to fill in prompts and generate code during validation. No "agentic loop" timing is reported, but the entire system – once assembled – runs autonomously. Prototyping thus combined manual skill development with Claude-driven code generation during testing.

**Hardware/Software Stack:** Implementation is open-source (GitHub). The tools run on Python 3.11+: key libraries include pandas, pyreadstat (for SAS files), and openpyxl. ClinAgent also uses *MCP SDK* to create Python/JS APIs (they mention Python & TypeScript SDKs). Claude Code integration was done via the Claude Code web IDE/API (Enterprise/Pro tier). The system was tested on a standard research laptop (no mention of GPU usage).

**Data & Provenance:** Validation used a real Phase 2 cardiovascular trial's specifications (metadata templates and ADaM specs) and *synthetic datasets* (generated by the Python Faker library) matching the trial's structure. There were 11 ADaM domains with 93,239 observations (synthetic). These stand in for real patient data, with no PHI involved. Input specs and program code were from the actual study ("STUDY-A"). Hence data provenance is anchored to a real trial design, but all values are fictitious.

**Validation & Performance:** ClinAgent's skills were rigorously tested. *Deterministic components* (log analysis, data validation) were perfect: log-analyzer found exactly 1 error and 7 warnings (100% precision), and data validator matched all 56 variables correctly. *Prompt-based tasks* (specification generation for tables) achieved 72.1% overall accuracy, with simple domains >96%. The authors estimate that this "derivation accuracy" is far above the ~30–50% achievable by generic LLMs without domain guidance. In practice, 72% of the generated specs were correct or nearly so. These metrics are industry-leading for trial automation.

**Regulatory/Ethical Status:** No patient data were used (only synthetic). All generated code is CDISC-compliant by design (ClinAgent embeds CDISC rules). The study did not seek IRB since no humans were involved. ClinAgent aims to support GxP-compliance by enforcing audit trails and versioning for any output. It is not itself regulated software; rather, it helps produce regulatory-grade deliverables (e.g. eSubmissions).

**Deployment & Use:** ClinAgent is published under an open-source license (MIT) and available on GitHub. It is not an off-the-shelf product yet, but the code and docs are public. A clinical programmer could deploy it today to automate TLF generation. No healthcare system has formally adopted it (it's a research prototype).

**Limitations & Failures:** The main limitation is the 72% accuracy on spec generation: about 28% of outputs still need manual correction. The authors caution that AI output remains a starting point requiring expert review. They also note that ClinAgent currently automates programming, not higher-level protocol writing. Failure modes include misaligned specs for complex domains (where Claude's prompting may miss niche rules). However, by separating data I/O (MCP tools) from logic, many errors are caught deterministically.

**Implications:** ClinAgent exemplifies how Claude Code can be embedded within clinical workflows via domain-aware tools. For oncology trials, this means that once a trial's schemas are defined, an AI agent can generate the bulk of analysis programs automatically, slashing the usual 12–24 FTE-month effort. In a physical AI trial context, ClinAgent could be used to process trial outputs (robot or otherwise) into submission-ready tables with minimal human intervention. Its modular design (skills + MCP) is especially suited to regulated environments because it ensures traceability.
