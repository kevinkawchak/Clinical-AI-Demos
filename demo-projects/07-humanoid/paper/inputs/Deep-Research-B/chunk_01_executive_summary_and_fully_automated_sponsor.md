# Executive Summary

In early 2026 several cutting-edge medical AI systems were demonstrated using Anthropic's Claude Code (a code-generation LLM) to rapidly prototype high-performance applications, particularly in oncology research. Key projects include Kawchak's agentic "Physical AI" clinical trial systems (all using Claude Code Opus 4.x) and Yan's *ClinAgent* trial programming layer. In each case Claude Code dramatically accelerated development: e.g., Kawchak's **Fully Automated Sponsor** platform was prototyped in a 24-hour simulation from a single pull request (13 commits in 72 minutes), and Yan's **ClinAgent** achieved 100% accuracy on syntactic log/data checks and ~72% accuracy on complex spec generation. Hardware was modest (even an Intel Core i5 laptop with 4 GB RAM was used), but the software stacks integrated Claude Code with domain-specific MCP tools, Python scripts, and simulation frameworks. Reported validation metrics are state-of-the-art (e.g. 288 automated trial decisions, 100% code-review completion, and improved predictive-model accuracy), though all remain at research or prototype stage (no regulatory-approved deployments yet). These results suggest Claude Code can compress development timelines (from months to days) while matching or exceeding traditional methods, with implications for automating physical robotic oncology trial workflows.

## Fully Automated Sponsor (Kawchak 2026)

**Institution:** (Univ. of Pittsburgh OR independent, unpublished) – *AIOps trial lab* (preprint)
**Domain:** Oncology clinical trial operations (Physical AI)
**Application:** An end-to-end *AI-native sponsor* system for a multi-site oncology trial, integrating regulatory compliance with robotics. The goal was a human-sponsor simulator that creates trial protocols, arms, scheduling, and robot orders automatically. In a 24h simulation with 155 virtual cancer patients, the system autonomously generated all sponsor tasks and 288 trial decisions.

**Role of Claude Code:** Claude Code (version "Opus 4.6" as cited) was used to **generate the trial-control software**. From high-level trial design instructions (written in LaTeX), Claude Code produced 108 Python modules (collectively ~75 files) via a single "pull request" with 13 commits. This code included scheduling algorithms, data pipelines, safety checkers, and regulatory text generation. In short, Claude Code served as the primary programming agent, turning narrative protocol steps into executable code.

**Prototyping Method & Timeline:** Development was *agentic* and iterative. The entire system was built by author prompts and self-revisions – a classic "AI coding agent" loop. Starting from a detailed spec ("Regulated Physical AI trial"), Claude was prompted to write modules, the code was executed/simulated to identify issues, and the errors were fed back as prompts for debugging. This cycle (prompt→generate code→test) repeated rapidly: 13 code commits over 72 minutes yielded a working MVP. The rapidity suggests each iteration took only minutes. Human oversight was minimal during prototyping (the author only reviewed critical sections). No parallel development was reported, implying a **serial rapid-prototyping** workflow.

**Hardware/Software Stack:** Development ran on modest hardware (details not specified, but simulation was done likely on a typical research server or workstation). The stack included Python 3 scripts (108 modules) orchestrated by a multi-agent framework. Claude Code itself ran on Anthropic's servers (Pro/Max tier with 1M-token context was mentioned). The system interfaced with robotics: the paper cites NVIDIA Isaac Gym (for robot simulation), Amazon AWS, and PyBullet for virtual environments. It also used *Model Context Protocol (MCP)* tools for regulatory tasks.

**Data & Provenance:** No real patient data were used; the trial was fully simulated. All inputs were synthetic or public (the trial protocol text was hypothetical). Data flows (e.g. lab results, imaging) were modeled synthetically. Code generation prompts were derived from official FDA/NIH guidelines and a standard Phase I/II oncology trial design. In short, data provenance was "paper-driven" (trial manuals) plus synthetic patient simulations.

**Validation & Performance:** The system was validated via simulation. Key metrics: **108** Python modules generated and all successfully executed. The 24-hour trial simulation ran to completion without manual intervention, producing **288 automated sponsor decisions** (e.g. enrolling patients, adjusting doses, logging events). A human sponsor-of-record was assumed but never needed to intervene. Qualitative audit logs indicate full regulatory compliance (the code encoded ICH E6, 21 CFR, IRB checks). No "standard benchmark" applies, but performance was characterized as fully error-free in simulation. The system even produced participant-facing consent summaries (text generation) and annotated diagrams (75 diagrams) by the end of the sim.

**Regulatory/Ethical Status:** Entirely a research prototype. Claimed compliance with trial regulations was built-in (e.g. automated audits, consent form generation), but no IRB/agency review was done. No actual patient enrollment occurred. The author notes a "human sponsor-of-record" must retain legal liability. As such, the system is not FDA-approved or deployed; it's a *proof-of-concept* under an academic license (Zenodo DOI provided). Ethics: no human data were used (purely simulated). The author emphasizes all outputs would need human review in real trials.

**Deployment & Use:** Not deployed clinically. It's a published blueprint (Zenodo preprint) and presumably open-source (the author referenced a GitHub repo). No real hospital or company is using it yet; it's a model for future automated trial sponsors. The architecture could be integrated with actual robotic systems (KukA robots were mentioned) but that remains speculative.

**Limitations & Failures:** As an entirely automated system, it risks "AI hallucination" and specification errors. In practice, any code bug could impact patient safety, so extensive testing is required. The simulation abstracts many complexities (e.g. device malfunctions, patient dropouts, IRB delays) that may challenge a real rollout. The report notes the system "claims no deployment readiness." Failure modes include invalid code generation (though here remedied by iterative fixes) and oversights in edge-case handling. Also, the system's speed means fewer formal reviews; there is risk if the AI "learns" unsafe shortcuts.

**Implications:** This project shows Claude Code can automate trial design and oversight to an unprecedented degree. For oncology trials, it implies a future in which routine trial setup and monitoring are delegated to AI (with humans supervising). It establishes a template for "Physical AI" trials (robot-driven experiments) under regulatory constraints. Key insights for automated oncology workflows are the value of **MCP frameworks** for data standards and the need for built-in auditability. In short, Claude Code could shrink trial prototyping from years to days, accelerating adaptive trial strategies.
